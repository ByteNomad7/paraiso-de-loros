---
name: sitemap.xml drift from split sitemaps
description: Why sitemap.xml (the file submitted to Google Search Console) can undercount real pages, and how to check/fix it.
---

The site has two parallel sitemap mechanisms that can disagree:
1. `sitemap.xml` — a flat file some older `generate-phase*.js` scripts appended to directly via a `track()` helper.
2. Split per-category files (`sitemap_main.xml`, `sitemap_blog.xml`, `sitemap_cities.xml`, `sitemap_species.xml`, `sitemap_fr.xml`, `sitemap_pt.xml`, `sitemap_accessories.xml`, `sitemap_en.xml`) generated by `generate-split-sitemaps.js` and referenced by `sitemap_index.xml` (which `robots.txt` correctly points to).

**Why:** Newer page batches (e.g. the `conocimiento/` and `preguntas/` FAQ hubs) were never registered in either mechanism, and `sitemap.xml` fell far behind the split sitemaps (602 vs. 865 real indexable URLs) because nothing kept them in sync. If GSC has `sitemap.xml` submitted directly (not `sitemap_index.xml`), it will only see the stale flat file.

**How to apply:** Before trusting sitemap coverage, diff the live `*.html` tree against `sitemap.xml`'s `<loc>` entries (normalize by stripping `index.html`/`.html`/trailing slash) and check any gap for a `noindex` meta tag — pages with `noindex` (404.html, gracias.html, a few retired blog posts) are correctly excluded; anything else missing should be added. A follow-up task (parent: "Set up the imported project") proposes unifying the two mechanisms — check its status before redoing this analysis.
